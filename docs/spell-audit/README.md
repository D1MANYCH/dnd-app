# Аудит SPELL_DATABASE против dnd.su — методология

Сверка `spells.js` → `SPELLS_BASE` (712 записей: 352 PH14 + 360 PH24) с авторитетным
RU-источником **dnd.su** по соответствующей редакции. План фаз и контекст —
`memory/project_spell_audit_plan.md` (триггер «начать фазу SPELL-AUDIT-N»).

## Харнесс

| Часть | Чем делается |
|---|---|
| Выборка записей БД + скелет отчёта | `node tools/spell-audit.js report <level> [PH14\|PH24]` → `docs/spell-audit/L<level>_<src>.md` |
| Что ещё нужно зарезолвить в URL | `node tools/spell-audit.js urls <level> [PH14\|PH24]` → готовые поисковые запросы |
| Распределение по уровням/редакциям | `node tools/spell-audit.js stats` |
| Проверка нормализации поля | `node tools/spell-audit.js norm "<строка>"` |
| RU-имя → URL | агент: `WebSearch "site:dnd.su <имя>"` → кэш `tests/_spell-url-map.json` |
| Чтение карточки заклинания | агент: `WebFetch <url>` (страница спелла рендерится полностью) |

Резолв URL и чтение страниц **нельзя заскриптовать** (динамический список + внешние
запросы) — это делает агент. Скрипт готовит вход (скелет с «нашей» стороной) и держит кэш.

### Edition-routing
- **PH14** (2014): `https://dnd.su/spells/<id>-<slug>/`
- **PH24** (2024): `https://next.dnd.su/spells/<id>-<slug>/` — отдельный поддомен `next.`!
- Один `WebSearch "site:dnd.su <имя>"` обычно отдаёт **оба** URL (dnd.su + next.dnd.su) — резолвим обе редакции за один поиск.
- Homebrew (`dnd.su/homebrew/...`) и бестиарий — **игнорировать**.

### Кэш URL (`tests/_spell-url-map.json`)
Ключ — имя из БД в нижнем регистре (ё→е). Значение — `{ en, ph14, ph24 }`. Пополняется
по мере резолва, чтобы не повторять поиск. 15-минутный кэш WebFetch — батчить, не гнать всё разом.

## Рубрика: формат vs ошибка

Сверяем поля `name, level, school, time, range, components, duration, classes, desc, higherLevel`.
**Различие ФОРМАТА — не ошибка.** Реальная ошибка — расхождение смысла/механики ПОСЛЕ нормализации.

| Поле | Нормализация (НЕ ошибка) | Ошибка (флагуем ⚠️) |
|---|---|---|
| **school** | регистр: наше `воплощение` = dnd.su `Воплощение`. Наши 8 школ = стандартные (карта EN в `tools/spell-audit.js`) | другая школа по сути |
| **range** | `120 фт` = `120 футов`; `На себя`/`Касание` | другое число/тип дистанции |
| **components** | латиница `V,S,M` = кириллица `В,С,М`; материал в скобках сравниваем по смыслу | отсутствует/лишний компонент; неверный материал |
| **duration** | `Мгновенно` = `Мгновенная`; `до` = `вплоть до` | **наличие/отсутствие концентрации**; другое время |
| **time** | `1 действие` = `1 действие`; `бонусное действие` | другой тип/число действий, ритуал |
| **classes** | порядок; перевод | отсутствует/лишний **PHB-класс** (см. скоуп ниже) |
| **name** | вариант перевода (см. РЕШЕНИЕ ниже) | другое заклинание / искажающая смысл ошибка |

### desc-механика (главное)
Наши `desc` — **намеренно сжатые пересказы**, НЕ дословный официальный текст. Поэтому сверка desc =
проверка **корректности механики**, а не дословности. Флагуем искажения и пропуск ключевой механики:

- кости и тип урона (`8к6 огнём`), число костей и масштабирование;
- спасбросок (характеристика) vs бросок атаки vs автопопадание;
- дистанция/радиус/форма области (`сфера 20 фт`);
- длительность и **концентрация**;
- условия, состояния, «половина при успехе», особые оговорки (укрытие, тип цели).

НЕ флагуем: парафраз, опущенный флавор, иной порядок слов, сокращения.

### ❓ Перепроверять вручную (анти-паттерны)
- **Сводка WebSearch врёт.** Доверяем только странице (WebFetch), не AI-аннотации в выдаче поиска.
  Пилот: поиск назвал Mage Hand «Evocation» — страница и канон дают **Conjuration** (наше «вызов» верно).
- **Фетч против канона.** Если WebFetch противоречит широко известной механике — флаг ❓, перепроверить.
  Пилот: фетч приписал Mage Hand «(с концентрацией)» — официально Mage Hand **без концентрации**, наше верно.

### Скоуп классов (PHB)
Приложение — про **PHB**. На dnd.su в списке классов 2014 встречается **изобретатель (artificer)** —
это класс из Tasha's/Eberron, не PHB. Его отсутствие в нашем `classes` — **намеренный скоуп, не ошибка**.
Флагуем только расхождения по PHB-классам. Для PH24-записей помним: 2024 официально **расширил**
классовые списки — это реальные отличия редакции, их и проверяем.

## РЕШЕНИЕ: имена — алиасы, не массовое переименование

**Держим имена БД каноничными; к dnd.su мостимся алиасами в `resolveSpellByName` (app-core.js).
Расхождение имени = «отметить, добавить алиас при отсутствии», НЕ правка. Переименовываем в БД
ТОЛЬКО если наше имя — настоящая ошибка (другое заклинание/искажающий смысл перевод).**

Почему так (а не массовый ренейм под dnd.su):
1. **Нулевой риск миграции.** `char.spells.mySpells` хранит **объекты-снимки** заклинаний (дедуп по `id`,
   `app-core.js`), каст ищет по `name` внутри собственного `mySpells` персонажа (`app-combat.js`),
   а `char.spells.prepared` — массив строк-имён. Ренейм в БД создал бы дрейф снимков и разъезд `prepared`.
2. **Билды не ломаются.** `character-builds.js` (headline-планы, `startingSpells`) и `build-notes-data.js`
   используют наши имена; массовый ренейм потребовал бы их синхронной правки.
3. **Инфраструктура алиасов уже есть и масштабируется** (`resolveSpellByName`, ~30 алиасов). Дёшево и обратимо.
4. **Имя dnd.su — часто вариант перевода, а не «единственно верное».** `Fire Bolt` = «Огненный болт»/«Огненный
   снаряд» — оба валидны. Флагать каждый вариант как ошибку = шум.
5. **Ценность аудита — корректность механики** (урон/спас/дистанция/длительность/школа/классы), она от имени не зависит.

Практика на каждое расхождение имени: убедиться, что имя dnd.su резолвится нашим `resolveSpellByName`
(через каноничное имя или алиас); если нет — **добавить алиас** `dnd.su-имя → наше-имя`. Это держит
билд-резолвер и поиск зелёными, без правки `name` в БД.

## Цикл фикса (фазы 1–6)
1. `report <level> <src>` → скелет. Резолвим URL (`urls`→ WebSearch → кэш), читаем страницы (WebFetch), заполняем «dnd.su» + «вердикт».
2. Собираем ⚠️ правки → правим **`spells.js`** (не отчёт). Имена — через алиасы в `app-core.js`.
3. Verify: `node tests/headless-node.js` + (если правили UI-видимое) preview рендера 2–3 спеллов.
4. **bump CACHE_NAME** в конце группы правок `spells.js` (иначе SW отдаёт старый код). Тулинг/доки (`tools/`, `docs/`) в кэш PWA не входят — их правки bump НЕ требуют.
