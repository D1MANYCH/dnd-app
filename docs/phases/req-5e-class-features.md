# REQ-5e — Аудит имён классовых умений (CLASS_FEATURES, 12 классов)

**Эталон:** книга **PHB 2014 «PHantom»** (`../DND rules/DnD5e Книга Игрока.pdf`, гл. 3 «Классы», стр. 47–119).
**Цель:** имена базовых классовых умений в `data.js` → `CLASS_FEATURES` привести к названиям книги.

## Область и безопасность
- В scope только **базовые** умения класса (`CLASS_FEATURES`). Умения подклассов (`SUBCLASS_FEATURES`) — это REQ-5d.
- `feature.name`/`feature.desc` — **только отображение** (`updateClassFeatures` в `app-combat.js` экранирует их в innerHTML; `openClassPlan` в `app-core.js`). Ключи = имя класса + уровень. **Миграция НЕ нужна** (фичи не хранятся в сейвах по имени).
- Generic-заглушки («Особенность пути/домена/архетипа/коллегии/клятвы/покровителя/происхождения/круга/школы») — это app-метки «получаете умение подкласса на этом уровне», у книги единого имени нет → **не трогаем**.
- Leveling-метки счётчиков («Три атаки», «Четыре атаки», «… 1к8/1к10») — UX-ярлыки прогрессии; книга переиспользует базовое имя. Оставляем, помечаем.
- **«Боевой стиль»/Fighting Styles — сквозной концепт** (Воин/Паладин/Следопыт + `class-choices.js` `FIGHTING_STYLES` + `CLASS_CHOICES`-заголовки). Делается целиком в одной партии, чтобы не было полупереименованного состояния.

## Метод сверки
Заголовки умений из книги извлечены через pymupdf по размеру шрифта: **13.0 pt = базовое умение класса**, 11.0 pt = умение подкласса (REQ-5d), 17.0 pt = заголовок секции подклассов. См. [[reference_dnd_rulebooks_pdfs]].

## Тип находок
- **rename** — app-умение механически = книжному, отличается только имя → переименовываем.
- **flag (content)** — app-умение по desc НЕ соответствует книжному умению этого уровня (выдумано/смещено/перепутано). **Вне scope REQ-5e (нейминг)** — это механика. Не переименовываем (книжное имя + неверный desc хуже), фиксируем для возможного отдельного «механического» прохода. Прецедент: REQ-5c оставил баг Ioun-Fortitude.
